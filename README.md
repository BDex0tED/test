# REST Service for Internal Transfers

Данный проект представляет собой REST-сервис для внутреннего перевода денежных средств между счетами внутри банка. Приложение построено с использованием слоистой архитектуры и обеспечивает транзакционную целостность, потокобезопасность и обработку ошибок.

## 🛠 Технологии
* **Java 17**
* **Spring Boot 3** (Web, Data JPA, Validation)
* **PostgreSQL** (в качестве основной БД)
* **Docker & Docker Compose**
* **JUnit 5 & MockMvc** (для тестирования)
* **Swagger/OpenAPI** (для документации)

## 🚀 Как запустить проект

```bash
docker-compose up --build
```


Приложение будет доступно по адресу: `http://localhost:8080`

## 🛡 Архитектурные решения

### Предотвращение Race Condition

Для предотвращения состояния гонки (race condition) при конкурентных переводах средств (например, когда два запроса одновременно пытаются списать деньги с одного и того же счета), в проекте применена **пессимистичная блокировка (Pessimistic Locking)**.

В слое репозитория (`BalanceRepo`) используется аннотация `@Lock(LockModeType.PESSIMISTIC_WRITE)` совместно с запросом `SELECT ... FOR UPDATE`.
Когда транзакция читает баланс для перевода, база данных блокирует эти строки. Любые другие параллельные транзакции, пытающиеся изменить эти же балансы, будут ожидать завершения первой транзакции. Это гарантирует отсутствие потерянных обновлений (lost updates) и сохраняет консистентность финансов.

### Транзакционность и Логирование

Все операции по изменению баланса выполняются строго внутри транзакции (`@Transactional`).
Логирование статусов транзакций (`SUCCESS` / `FAILED`) вынесено в отдельный компонент `TransactionLogger` с использованием `Propagation.REQUIRES_NEW`. Это позволяет гарантированно сохранять историю попыток переводов даже в случае, если основная транзакция перевода откатывается (Rollback) из-за бизнес-ошибки (например, нехватка средств).

### Идемпотентность (Дополнительное задание)

Для защиты от случайных повторных списаний (например, при проблемах с сетью на клиенте) реализована поддержка заголовка `Idempotency-Key`. Если клиент отправляет запрос с уже известным ключом, сервер не выполняет перевод повторно, а возвращает сохраненный результат предыдущей успешной операции.

```http
POST /api/transfers HTTP/1.1
Host: localhost:8080
Content-Type: application/json
Idempotency-Key: 123e4567-e89b-12d3-a456-426614174000

{
    "amount": 500.00,
    "fromBalanceId": 1,
    "toBalanceId": 2
}

```
